# 官方 HEX 回归执行计划

更新时间：2026-07-01

## 1. 目标与边界

本计划供新的 Codex 会话直接接续，用于建立官方 HEX 回归基础设施并运行当前允许清单。本阶段只验证现有乱序 Core，不实现 RV32F、Zba、Cache、MMIO、多特权等级或其他新功能。

开始官方回归前，必须先确认：

- 全量 SystemVerilog 编译为 `0 Errors, 0 Warnings`；
- 当前 27 项模块、单项指令及组合场景测试全部 `PASS`；
- `hex/riscv-tests/index.csv` 中每个条目均有对应 HEX，且不存在未索引 HEX；
- 工作区中的既有用户修改不得被覆盖或回退。

## 2. 当前 HEX 范围

当前 `hex/riscv-tests/` 保留 66 个物理地址版 HEX：

| 分组 | 数量 | 本阶段处理方式 |
|---|---:|---|
| RV32I (`rv32ui-p-*`) | 42 | 纳入回归，`ma_data` 暂跳过 |
| RV32M (`rv32um-p-*`) | 8 | 全部纳入回归 |
| RV32F (`rv32uf-p-*`) | 11 | 仅保留，不运行；当前 Core 未实现 F |
| Zicsr (`csr/mcsr`) | 2 | `csr` 纳入；`mcsr` 暂跳过 |
| Zba (`sh1add/sh2add/sh3add`) | 3 | 仅保留作未来 ALU 扩展实验，不运行 |

不得重新引入已删除的虚拟地址版、A/C/D/S-mode、Zfh 或复杂位操作/密码学扩展镜像。确需恢复时，必须由用户明确决定。

## 3. 回归基础设施

### 3.1 官方回归 testbench

新增独立的 Core 级官方回归 testbench，要求：

- `core_top.RESET_PC = 32'h8000_0000`；
- 统一行为内存同时服务 IMEM/DMEM，基地址为 `0x8000_0000`；
- 通过 `+HEX=<path>` 加载单个镜像；
- 乘法使用固定延迟行为模型，延迟必须与 `MUL_LATENCY` 一致；
- 除法使用双输入 ready/valid 行为模型，并覆盖除零和有符号溢出的既有本地快速路径；
- 保留提交总线、recovery、DMEM 请求和关键内部状态的只读观测；
- 不为通过测试而修改原始 HEX 内容。

### 3.2 PASS/FAIL 判定

首版沿用旧设计的架构结束约定：

1. 任一提交口提交 PC `0x8000_0044`；
2. 等待该提交沿完成；
3. 通过 `RRAT[x3] → PRF` 读取已提交 `gp/x3`；
4. `gp == 1` 判定为 PASS，其他值判定为 FAIL。

如果实际镜像并不使用该结束约定，必须先通过提交轨迹和镜像反汇编证明确切结束协议，再修改 testbench；不得通过猜测地址放宽判定。

默认超时为 100000 个周期。超时、断言、仿真非零退出、未知值污染或未出现结束事件均为失败。

### 3.3 运行入口

新增 PowerShell 主脚本，BAT 只允许作为兼容包装。脚本应支持：

- `unit`：现有 27 项测试；
- `official-smoke`：少量 RV32I 冒烟用例；
- `official-i`：RV32I 允许清单；
- `official-m`：全部 RV32M；
- `official-csr`：`rv32mi-p-csr`；
- `official`：所有当前允许的官方用例；
- `all`：编译、27 项测试和官方允许清单。

每项生成独立日志，汇总 PASS、FAIL、TIMEOUT、SKIP 和运行周期。日志及波形写入 Git 忽略的 `results/`。

## 4. 执行顺序

严格逐级执行，前一级未稳定通过时不得扩大范围：

1. 编译及 27 项既有回归；
2. `rv32ui-p-simple`、`add`、`addi` 冒烟；
3. RV32I ALU/移位/比较；
4. RV32I 分支与 JAL/JALR；
5. RV32I Load/Store 与 FENCE.I，跳过 `ma_data`；
6. RV32M 八项；
7. `rv32mi-p-csr`；
8. 汇总允许清单与跳过原因。

一次只处理一个首个失败用例。不得在前一个失败尚未闭环时并行修改多个功能点。

## 5. 失败处理硬规则

以下规则是本计划的强制约束，不得因为额度、时间或“可能有效”而跳过。

### 5.1 先排除测试基础设施问题

每个失败首先只做只读诊断，依次检查：

1. HEX 路径、索引条目、基地址、文件行数和 `$readmemh` 加载范围；
2. testbench 的复位、时钟、结束 PC、gp 读取层次和超时条件；
3. IMEM 一拍同步时序与 Core 外 DMEM 寄存级；
4. 乘法固定延迟、除法双通道握手及结果保持；
5. 失败是否可稳定复现，是否只由前序测试残留状态造成；
6. 测试要求是否超出当前实现范围。

只有确认测试文件、runner、行为模型和判定逻辑没有问题后，才可把失败归因于 RTL。

### 5.2 首次无法定位时立即交给用户查波形

第一次诊断如果不能得到明确、可证伪的根因假设，Codex 必须立即停止修改 RTL，不得盲试补丁。应向用户提交一份失败情况单，至少包含：

- 用例名、HEX 路径、随机性/可复现性；
- 首个异常周期和最后一个正常提交 PC；
- 期望行为与实际行为；
- 是否为 FAIL、TIMEOUT、异常 recovery 或数据错误；
- 已排除的 testbench/镜像因素；
- 建议保存的波形时间窗；
- 建议观察的信号及层次路径；
- 1～3 个尚未证实的可能原因，明确标记为“假设”。

随后由用户手动查看波形并给出线索。用户给出新证据前，Codex不得继续修改该问题。

### 5.3 Codex 修复次数最多三次

同一个首个失败用例，在排除测试问题并确认 RTL 根因方向后，Codex 最多进行三次修复尝试。

一次“修复尝试”定义为：根据一个明确假设修改 RTL 或架构可见控制逻辑，并运行目标用例验证。仅增加日志、断言、波形观测或修正 testbench 不计入 RTL 修复次数。

每次尝试必须记录：

- 尝试编号（1/3、2/3、3/3）；
- 根因假设和支持证据；
- 修改文件及逻辑范围；
- 目标用例结果；
- 27 项既有回归结果；
- 假设被证实或证伪的结论。

禁止在一次尝试中混入多个无关 RTL 改动，以规避次数限制。

第三次仍失败时必须停止：

- 不得进行第四次修复；
- 不得降低断言、延长超时或修改期望值来伪造通过；
- 保留当前日志、失败命令和波形；
- 汇报三次尝试的差异、现象变化和剩余假设；
- 等待用户手动查波形或决定是否重新设计。

### 5.4 每次修复后的验证门禁

目标用例通过后必须按以下顺序验证：

1. 同分组相关用例；
2. 全量编译 `0 Errors, 0 Warnings`；
3. 27 项既有回归全部通过；
4. 已经通过的官方用例不得回退。

任何一级失败都视为本次修复未闭环。

## 6. 波形交接建议

用户手动排查时，优先保存以下信号：

- IF/ID：取指 PC、IMEM 地址/数据、valid/allowin、预测信息；
- Rename：FIFO count、RAT/RRAT、Free List、Busy Table、源物理寄存器及 ready；
- Dispatch/ROB：双路分配、tag、head/tail、complete、commit；
- IQ/Operand：入队、wakeup、issue、PRF 读地址和旁路数据；
- LSQ/DMEM：occupancy、地址完成、转发选择、请求寄存级、response；
- Execute/WB：各执行单元 valid/ready、WB0/WB1 仲裁、广播；
- Control：branch update、recovery reason/target、CSR trap_event、mepc/mcause/mtval。

波形应至少覆盖“最后一个正常提交前 20 拍”到“首个错误现象后 20 拍”。若为超时，应截取首次停止前进的位置，而不是保存完整十万周期波形。

## 7. 完成标准

- 统一 runner 和清单可重复运行；
- 27 项既有测试保持全部通过；
- RV32I 允许清单、全部 RV32M 和 `rv32mi-p-csr` 全部通过；
- `ma_data`、RV32F、`mcsr`、Zba 均以明确原因标记 SKIP；
- 每个失败都遵守“三次修复上限”和“首次无法定位即交用户查波形”的规则；
- README、PLAN、WORKSPACE_HANDOFF 记录最终通过、失败和跳过清单。

本文件只定义下一会话的执行方案，不在当前会话启动官方 HEX 回归。
